Introduction Link to heading

This project did not start as an attempt to win a handwriting-recognition competition. The goal was more practical: build a small OCR system that can run close to the user, possibly even on a mobile phone, and measure how far such a system can go before larger models become necessary.

That changes the engineering priorities. A leaderboard solution can spend more memory, use heavier backbones, rely on large language models, or hide complexity behind expensive inference. A practical OCR system has different constraints:

  • It should be small enough to deploy without special hardware.
  • It should be fast enough for interactive use.
  • It should be trainable and debuggable by one person.
  • It should expose where mistakes come from instead of behaving like a black box.
  • It should provide a realistic estimate of quality under mobile-friendly constraints.

The central question is therefore not “how do we get the best possible CER at any cost?”, but:

How good can handwritten OCR become when the full pipeline is designed around lightweight models, transparent postprocessing, and deployable inference?

This article follows that question through the whole system: line detection, recognition, decoding, language-model rescoring, and document-level correction. The interesting part is not a single model checkpoint, but the way each stage changes the error profile of the next one.

The dataset contains full-page handwritten documents and extracted recognition crops with different levels of difficulty: clean lines, dense pages, noisy regions, and ambiguous crops. A few examples are enough to show why this is a pipeline problem rather than only a recognizer problem.

Problem Statement Link to heading

Handwritten OCR looks deceptively simple: detect text lines, recognize text, and measure CER. In practice, every time one bottleneck is solved, another becomes dominant somewhere else in the pipeline.

This article documents the engineering journey of building a complete handwritten OCR system using lightweight models and practical techniques rather than large foundation models.

Goals Link to heading

  • Build a complete handwritten OCR pipeline from page image to final text.
  • Keep the main components lightweight enough for local or mobile-oriented inference.
  • Understand the quality that is realistically achievable without large foundation models.
  • Make each stage measurable: detection errors, recognition errors, decoding errors, and language-model corrections.
  • Incrementally improve the system through targeted engineering rather than opaque end-to-end scaling.

Non-Goals Link to heading

  • Beating every public benchmark.
  • Optimizing for server-side batch inference.
  • Depending on a large multimodal or language model for the core OCR task.
  • Treating postprocessing as a hidden correction layer that cannot be inspected.

The intended outcome is a practical quality baseline: if the system fails, it should be clear whether the failure comes from line extraction, visual recognition, decoding, dictionary candidates, or document context. That is more useful for product-oriented OCR than a single aggregate score.

Starting From Detection Research Link to heading

The first design decision was how to extract text lines from a full page. The dataset looks like ordinary page images with line-level annotations, so the simplest option would be to treat every line as an object and train a YOLO-style detector, following the general bounding-box detection approach introduced in You Only Look Once: Unified, Real-Time Object Detection.

That is attractive because the tooling is mature:

  • annotation formats are simple;
  • training and evaluation pipelines are well known;
  • inference is fast;
  • mobile deployment is realistic.

But handwritten text lines are not natural rectangular objects. A bounding-box detector turns line extraction into a box prediction and filtering problem, and that creates several issues for this task:

  • Dense pages produce many overlapping boxes, so non-maximum suppression becomes part of the core algorithm instead of a minor cleanup step.
  • Curved lines are poorly represented by rectangles.
  • Neighboring lines can touch or overlap vertically, making box boundaries ambiguous.
  • A box can localize a line while still producing a bad crop for recognition.
  • The detector does not naturally express separators between adjacent lines.

For this project, those tradeoffs felt wrong. The line detector should describe the geometry of handwriting, not force it into rectangular proposals. That led to a segmentation-based approach inspired by A Two-Stage Method for Text Line Detection in Historical Documents.

The key idea in that paper is to predict pixel-level classes instead of boxes. The first stage uses ARU-Net to classify pixels as baseline, separator, or background. The second stage converts those predictions into line hypotheses using image-processing logic. This is a better fit for historical and handwritten documents because the representation can follow curved or arbitrarily oriented text lines, and separators provide an explicit signal for splitting dense regions.

I adapted that direction, but with a slightly different practical goal: not just detecting baselines for evaluation, but producing stable line crops for a downstream recognizer. In other words, detection quality is judged by the recognition pipeline it enables.

The figure below shows the type of signal the detector is expected to learn. The input is a dense handwritten page, while the model predicts two complementary maps: a line-probability map that highlights text centerlines and a separator map that marks barriers between text regions.

The line-probability map replaces rectangular proposals with a soft centerline signal, while the separator map gives postprocessing an explicit cue for splitting nearby text regions. This is the part that makes the method useful on dense pages where bounding boxes and connected components tend to become unstable.

Part 1. Building the Core OCR System Link to heading

1. Line Detection Link to heading

Why Line Detection Matters Link to heading

Many OCR systems focus primarily on recognition. In practice, line extraction often determines the upper bound of achievable accuracy.

Challenges:

  • Curved handwriting
  • Variable line spacing
  • Touching lines
  • Overlapping strokes
  • Inconsistent layouts

Target Design Link to heading

Instead of predicting bounding boxes, the model predicts:

  • Text centerlines
  • Separator maps

Motivation:

  • More robust to curved text
  • Easier separation of touching lines
  • Better handling of variable spacing

ARUNet Link to heading

For line detection I used an ARUNet-style segmentation model. Conceptually, it is a U-Net encoder-decoder network with attention blocks added to improve spatial focus. The encoder extracts increasingly abstract page features, while the decoder upsamples them back to pixel-level prediction maps.

The important part is that the model remains fully convolutional: it predicts dense maps over the whole page instead of proposing individual boxes. Spatial attention helps the decoder focus on relevant text-line structures while suppressing background texture, page noise, and unrelated strokes.

1Page Image
2    ↓
3Encoder
4    ↓
5Spatial Attention
6    ↓
7Decoder
8    ↓
9Centerline Map + Separator Map

The training targets are not rectangular boxes. They are dense supervision maps: one map describes likely text centerlines, and another describes separator regions. This representation is what makes the later superpixel and connected-component postprocessing possible.

V0 Postprocessing: From Probability Maps to Lines Link to heading

The network output is not a set of text lines. It is only a pair of probability maps:

  • a baseline or centerline map;
  • a separator map.

Those maps are useful for visualization, but they cannot be passed directly to the recognizer. The recognizer needs ordered line crops: one crop per text line, with enough vertical margin to preserve ascenders and descenders, but not so much that neighboring lines leak into the crop.

This is where postprocessing becomes a real part of the OCR system rather than a small cleanup step. I partially followed the logic from A Two-Stage Method for Text Line Detection in Historical Documents: use the neural network to produce soft geometric evidence, then use classical image-processing and graph logic to recover line instances.

The first version of the postprocessor lives around the following idea:

 1Baseline Probability Map
 2    ↓
 3Threshold + Skeletonization
 4    ↓
 5Sparse Superpixels
 6    ↓
 7Local Orientation + Interline Spacing
 8    ↓
 9Candidate Links
10    ↓
11Connected Components
12    ↓
13Curve Fitting
14    ↓
15Line Boxes

The important change compared with a bounding-box detector is that the algorithm does not ask the neural network to solve the entire grouping problem. The model predicts where line centers and separators are likely to be; postprocessing turns that soft evidence into geometry.

Superpixels Link to heading

The first step is to reduce the dense baseline map to a sparse set of points. The baseline map is thresholded, skeletonized, and then sampled into superpixels. Each superpixel is a small representative point on a likely text centerline.

Sampling is confidence-ordered: points with higher baseline probability are considered first. A minimum-distance rule prevents the skeleton from producing too many nearly identical points. This gives a compact graph problem instead of a full-resolution pixel problem.

Conceptually:

1for point in skeleton_points_sorted_by_confidence:
2    if point is far enough from already selected points:
3        keep point as a superpixel

This step is important for speed. It keeps the postprocessor lightweight enough for local inference while preserving the structure of the page.

Local State Estimation Link to heading

A superpixel alone is only a point. To decide whether two points belong to the same line, the algorithm also estimates local state:

  • local text orientation;
  • approximate interline distance;
  • neighborhood connectivity.

The implementation builds a Delaunay graph over the superpixels and measures how strongly neighboring points are connected through the baseline probability map. Local orientation is estimated from the best-connected neighbors. Interline spacing is estimated from projection profiles around each point; when graph-cut support is available, spacing labels are smoothed over the graph.

The reason for estimating spacing is that a fixed pixel threshold is brittle. A page can contain different resolutions, writing sizes, margins, and line gaps. Normalizing decisions by local interline distance makes the same postprocessor work across more pages.

Linking Superpixels Link to heading

After every superpixel has a position, orientation, and spacing estimate, the algorithm builds candidate links between points that plausibly continue the same text line.

A candidate link must satisfy several local constraints:

  • it must move forward along the text direction;
  • the horizontal gap cannot be too large;
  • the perpendicular offset must be small relative to the estimated line spacing;
  • the orientation difference must be limited;
  • the segment should stay close to horizontal for normal page text;
  • there should not be a better intervening candidate between the two points.

This produces a sparse graph where edges mean “these two superpixels probably belong to the same baseline.” Connected components of that graph become initial line fragments.

Iterative Merging Link to heading

Initial connected components are not always complete lines. A line can be broken when the ink is faint, when the baseline probability map has gaps, or when the local linking constraints are conservative.

To recover longer lines, the implementation grows clusters iteratively:

  1. Sort clusters by size and use larger clusters as seeds.
  2. Fit a smooth curve through the seed cluster.
  3. Search for nearby candidate clusters.
  4. Reject merges that do not have enough baseline support between cluster endpoints.
  5. Reject merges that make the fitted curve too noisy or too curved.
  6. Repeat until no useful merges remain.

This gives a useful balance. The first connected-component pass is strict, so it avoids many wrong merges. The iterative curve-fitting pass can then reconnect fragments when the geometry is consistent.

From Baselines to Recognition Boxes Link to heading

The final recognizer cannot read a one-pixel baseline. It needs an image crop containing the full handwriting around that line. This means that the last step of detection is not only “find a line”, but “produce a crop that is good for recognition.”

Each connected-component cluster is first converted into a smooth baseline. For clusters with enough points, the implementation fits a low-degree curve; for very small clusters, it keeps the original points. This gives a continuous line geometry:

1Cluster Points
2    ↓
3Ordered Points
4    ↓
5Curve Fit
6    ↓
7Sampled Baseline

The baseline is then converted into an oriented quadrilateral. The horizontal extent comes from the minimum and maximum x-coordinates of the baseline, with a small point margin on both sides. The vertical extent comes from local interline spacing:

1half_height = max(0.5 * local_spacing, 0.5 * min_line_box_height)

So the raw oriented box is built around the baseline:

1top-left     = (x_min, y_left  - half_height)
2top-right    = (x_max, y_right - half_height)
3bottom-right = (x_max, y_right + half_height)
4bottom-left  = (x_min, y_left  + half_height)

This is already better than an axis-aligned bounding box because the crop follows the local text direction. Even if the line is slightly slanted, the recognizer receives a crop aligned with the handwriting instead of a larger rectangular region full of background.

The separator map is used to improve the horizontal extent. If separator runs indicate that the text band continues slightly beyond the cluster endpoints, the box can be extended left or right. This is useful when the baseline cluster misses a faint beginning or ending of the line.

After raw boxes are created, the postprocessor performs two cleanup steps.

First, boxes that clearly describe the same line fragment are merged. The merge check is conservative: boxes need compatible centerlines and enough vertical overlap. This prevents adjacent lines from being merged just because their x-ranges overlap.

Second, boxes are expanded for recognition. The recognizer benefits from a little extra context around the text, especially for tall ascenders, descenders, and slightly inaccurate baselines. But expansion is limited by an IoU budget with neighboring lines. If two boxes already overlap vertically, the algorithm pre-clips them and only allows extra padding when the resulting overlap stays below the configured threshold.

The output of postprocessing is therefore two related objects:

  • raw oriented boxes, useful for scoring and debugging;
  • expanded recognition boxes, useful for the OCR recognizer.

This last step is where detection becomes directly tied to recognition quality. A crop that is too tight loses strokes. A crop that is too large introduces text from adjacent lines. The postprocessor has to find a practical middle point, and doing it explicitly makes failures inspectable.

Advantages of the v0 postprocessing style:

  • Robust separation of neighboring lines
  • Stable extraction of line regions from sparse geometry
  • Better handling of curved or fragmented baselines
  • Easier debugging than direct dense-map-to-box conversion

2. Recognition Link to heading

Baseline: CRNN + CTC Link to heading

After the detector started producing usable line crops, the next question was recognition. For the first baseline I wanted something deliberately simple: a model that is easy to train, easy to debug, and fast enough to run locally.

I used the setup described in Best Practices for a Handwritten Text Recognition System as the starting point: a CRNN-style recognizer trained with CTC loss. The point was not to build the strongest possible recognizer immediately, but to create a stable baseline that could tell whether the line detector was good enough.

The baseline recognizer has three main parts:

1CNN
2 ↓
3Sequence Features
4 ↓
5BiLSTM
6 ↓
7Character Logits
8 ↓
9CTC Loss

The CNN extracts visual features from the normalized line crop. The feature map is then converted into a left-to-right sequence, usually by collapsing the vertical dimension and treating the horizontal dimension as time. The recurrent layers model context along the line, and the final classifier predicts character probabilities for each time step.

CTC is a natural fit here because the training data gives the final text string, not character-level positions. The model is allowed to learn the alignment between image columns and output characters by itself:

1image columns:  t1  t2  t3  ...  tn
2CTC states:     blank, letters, repeats, blank
3output text:    decoded after collapsing repeats and blanks

This makes the recognizer much easier to train than a fully supervised character-segmentation model.

Auxiliary CTC Head Link to heading

One useful detail from the best-practices paper is the auxiliary head. Instead of training only the final recurrent output, the model also attaches an auxiliary CTC classifier to an earlier sequence representation.

Conceptually:

1CNN Features
2    ├── Auxiliary CTC Head
3    ↓
4BiLSTM
5    ↓
6Main CTC Head

The auxiliary branch acts like an optimization shortcut. It pushes the visual backbone to produce features that are already useful for character recognition before the recurrent layers refine them. The final loss is a weighted combination:

1loss = main_ctc_loss + λ * auxiliary_ctc_loss

This is useful in a practical OCR pipeline because it makes early experiments more stable. If the auxiliary head learns something but the final head does not, the issue is likely in sequence modeling or training configuration. If neither learns, the problem is more likely data, crops, normalization, or labels.

Attention Over the LSTM Sequence Link to heading

The baseline model also has an optional attention layer after the BiLSTM. In the code this is implemented as SequenceSelfAttentionBlock: a multi-head self-attention block followed by a feed-forward layer, both with residual connections and layer normalization.

The shape of the recognizer then becomes:

1CNN Features
2    ↓
3BiLSTM
4    ↓
5Sequence Self-Attention
6    ↓
7Main CTC Head

The BiLSTM already gives each timestep left and right context, but it still processes the line through recurrent state. The self-attention layer gives every output position a more direct way to compare itself with distant positions in the same line. That can help with long words, repeated shapes, and ambiguous local strokes where the correct character depends on wider context.

This was a conservative addition: it does not change the CTC training objective or decoding interface. If enabled, it only refines the encoder sequence before the final linear CTC classifier. That made it easy to compare against the plain CRNN baseline without changing the rest of the pipeline.

Single-Head Recognition Link to heading

The simplest version of the recognizer uses one output vocabulary for everything. All line crops go through the same CNN, the same recurrent encoder, and the same CTC head:

1Line Crop
2    ↓
3Shared CNN
4    ↓
5Shared BiLSTM
6    ↓
7Single CTC Head
8    ↓
9Text

This is the most convenient setup because there is only one character set, one decoder, and one checkpoint. It is also the easiest model to deploy: every input line is treated the same way.

The tradeoff is that one head has to model all content types at once. In my dataset, not every crop has the same distribution. Some regions are ordinary handwriting, some are printed text, and some may contain formula-like content or symbols. A single vocabulary can cover all of them, but it forces one classifier to learn a mixed distribution:

1P(character | image, all region types)

That can be useful when the region type is unknown, but it also makes the model spend capacity on rare symbols and domain-specific cases that may not help ordinary handwritten lines.

Additional Region Classification Head Link to heading

To make the single-head model more informative, I added an optional region classification head. In the code this is region_type_head: it pools the CNN feature map and predicts the type of the input crop.

1CNN Features
2    ├── CTC Head → text
3    └── Region Type Head → handwritten / printed / formula / ...

The training loss becomes:

1loss =
2    main_ctc_loss
3  + λ_aux * auxiliary_ctc_loss
4  + λ_region * region_type_loss

This head is not used to transcribe characters directly. Its purpose is diagnostic and regularizing:

  • it checks whether the visual backbone can distinguish region types;
  • it gives a useful signal when errors are concentrated in one content type;
  • it can later be used for routing to specialized recognizers;
  • it encourages pooled CNN features to preserve global information about the line crop.

This is especially useful when the OCR pipeline may receive different kinds of regions from the detector. Recognition errors are easier to understand if the model can also say, “this looks like printed text” or “this looks like formula content.”

Multi-Head Recognition Link to heading

The next experiment was a multi-head recognizer. Instead of one CTC vocabulary for all content, the model shares the visual backbone but uses a separate CTC head for each region type:

1Line Crop
2    ↓
3Shared CNN
4    ↓
5Head[handwritten] → handwritten charset
6Head[printed]     → printed charset
7Head[formula]     → formula charset

Each head has its own primary CTC logits and its own auxiliary CTC logits. During training, a batch from a particular region type updates the matching head:

1loss_for_handwritten =
2    CTC(head_handwritten)
3  + λ_aux * auxiliary_CTC(head_handwritten)
4  + λ_compat * compatibility_loss

The benefit is specialization. The handwritten head does not need to reserve output classes for symbols that only appear in formulas, and the formula head can use a vocabulary that better matches formula-like content. This is closer to training several recognizers, but with a shared CNN backbone.

The multi-head model also has compatibility heads. Each head predicts whether it is a good match for the input crop:

1compatibility_head[handwritten](features) → score
2compatibility_head[printed](features)     → score
3compatibility_head[formula](features)     → score

At evaluation time, routing can combine two signals:

  • the compatibility score for each head;
  • a simple confidence score from that head’s CTC logits.

Conceptually:

1routing_score(head) =
2    compatibility_score(head)
3  + α * ctc_confidence(head)

This gives two evaluation modes. In oracle mode, the known region type selects the correct head, which measures how good each specialized recognizer can be. In routed mode, the model chooses the head itself, which measures whether the system can work without external region labels.

The cost is deployment complexity. Multi-head recognition needs multiple vocabularies, multiple decoders, and routing logic. For a lightweight mobile-oriented OCR system, that tradeoff only makes sense if specialization gives a clear quality improvement.

Problem: CTC Had Too Few Time Steps Link to heading

The first serious debugging loop started with printed text. Based on visual inspection, these lines looked easier than handwriting: the glyphs were clean, the spacing was regular, and the crops were usually readable. But the validation metrics showed many errors for printed regions.

At first this looked suspicious. If the model fails on messy handwriting, there are many possible explanations: bad crops, ambiguous letters, inconsistent labels, or weak language modeling. But when it fails on clean printed text and produces complete nonsense, the problem is often more structural.

The debugging process was:

  1. Split metrics by region type.
  2. Notice that printed text had unexpectedly bad errors.
  3. Manually inspect printed validation crops.
  4. Confirm that many failed crops were visually clear.
  5. Compare the crop width with the target text length.
  6. Check whether the CTC sequence had enough timesteps to represent the label.

The key observation was that printed text is visually compact. A printed line can contain many characters in a relatively short crop. After resizing, the model may see a clean image, but the CNN may compress the horizontal axis so much that the CTC head receives too few sequence positions.

CTC needs enough input steps to align the output string. It is not enough that the image is readable to a human. The model needs a time dimension long enough to place characters, repeated characters, and blanks.

In the original geometry, the CNN reduced width by roughly 16:

1legacy_w16:
2input width → width / 16 sequence steps

For handwriting this was often acceptable because handwritten text tends to occupy more horizontal space per character. Printed text can be much denser. The same resized width can correspond to a much longer label.

To make this measurable, I added a simple CTC geometry check:

1ctc_ratio = sequence_length / label_length

The check estimates how many sequence steps the recognizer will produce for a given resized crop width, then compares that value with the length of the target label.

Then I added dataset-level logging for printed samples. For each printed crop, the loader records:

1file_name
2label_len
3resized_width
4ctc_length
5ctc_ratio
6warned
7filtered

This turned the issue from “the model is bad on printed text” into a concrete geometric question: how much temporal resolution does CTC have for each label?

The important cases were the low-ratio examples. If ctc_length < label_len, the target is impossible to align. Even above that threshold, the model can still be under pressure because CTC also uses blank symbols and has difficulty with repeated characters. With zero_infinity=True, impossible samples do not necessarily destroy training, but they are still a sign that the architecture is not giving CTC enough room.

The debug summaries and CSV files made this visible:

1printed CTC ratio:
2samples=...
3warned=...
4filtered=...
5impossible=...
6mean_T=...
7mean_ratio=...
8p10=...
9p25=...

After sorting by ctc_ratio, the worst examples matched the manual failure cases: clear printed crops with long labels and too few CTC timesteps.

The fix was to add a denser CNN geometry:

1dense_w8:
2input width → width / 8 sequence steps

The required code change was to make the recognizer geometry explicit. I kept two supported variants:

1legacy_w16
2dense_w8

In the CNN, the legacy variant uses horizontal stride in the first convolution. The dense variant removes that early horizontal downsampling and keeps more width resolution before later pooling. That roughly doubles the number of CTC timesteps without changing the high-level recognizer interface.

The training configuration then has to use the same cnn_variant everywhere:

  • model construction;
  • sequence-length calculation for CTC loss;
  • dataset CTC-ratio auditing;
  • attention-decoder or Conformer training if the checkpoint is reused.

This part is important. If the model produces dense_w8 features but the loss still computes input lengths as if the model were legacy_w16, CTC will be trained with the wrong sequence lengths.

This improved the situation because printed text needs more temporal resolution. The lesson was that CTC recognizers are not only about model capacity. The geometry of the CNN encoder determines whether the loss has enough sequence length to express the target text at all.

Problem: Label Noise and Bad Crops Link to heading

The next issue was not architectural. It was data quality.

For OCR datasets, it is easy to assume that the labels are correct because they came from an annotation pipeline. In practice, this assumption is dangerous. A recognizer trained on noisy labels can look like a weak model even when the architecture is fine.

I treated this as an active-learning cleanup problem:

1Train baseline recognizer
2    ↓
3Evaluate training samples
4    ↓
5Find suspicious samples by metrics
6    ↓
7Inspect and clean labels/crops
8    ↓
9Retrain recognizer

The idea is simple: once the model is good enough, its largest training-set errors are often not “hard examples” anymore. Many of them are annotation problems, broken crops, or samples whose image does not match the target text.

For the check step, I used several metrics:

  • CER: normalized character error rate between prediction and label.
  • Levenshtein distance: absolute edit distance, useful for catching large mismatches on longer lines.
  • Prediction confidence: low-confidence predictions are useful candidates for inspection.
  • Region-type classifier confidence: high-confidence disagreement between expected and predicted type can reveal mislabeled regions.

The main filters were:

  • high CER;
  • high Levenshtein distance;
  • high classifier confidence for a different region type;
  • low recognizer confidence combined with a large text mismatch.

After sorting candidates by those signals and opening the images manually, several problems became obvious.

Some crops contained more than one text line. In those cases the model had no principled way to know which line the label referred to. The image might contain two readable lines, while the target contains only one of them:

1crop image:  line A + line B
2target:      line A
3model:       line B, or a mixture of both

Those samples are especially harmful because the visual input and label are both valid text, but they do not describe the same recognition task.

This is particularly bad for CTC training. CTC assumes that the target string is a left-to-right sequence generated by the visual sequence in the input crop. If the crop contains two lines, the image no longer has a single obvious target sequence.

For example:

1crop:   upper line + lower line
2label:  lower line
3model:  may focus on the upper line

From the model’s point of view, the upper line can be clearer, longer, or more visually dominant. CTC will still force the network to align the provided label somewhere across the whole crop. That teaches the wrong association: visual features from one line are optimized toward text from another line.

The result is noisy supervision. The model is penalized for reading text that is actually present in the image, and rewarded for ignoring part of the crop. Removing these multiline samples made the training objective more consistent.

Other crops were simply messy: partial words, broken extraction, wrong region boundaries, or fragments where even a human would need surrounding context. Keeping those samples in training teaches the model inconsistent alignments.

I also used the classifier head to find region-type mismatches. For example, if a crop is labeled as handwritten but the classifier is highly confident that it is printed or formula-like, that sample becomes a review candidate.

The important part is that removal was not fully automatic. Metrics were used to rank suspicious samples, then the final decision was made by visual inspection. Multiline crops, broken crops, and clear wrong-label examples were moved out of the training set. After cleanup, the recognizer was trained again on the refined dataset.

Why This Was a Good First Baseline Link to heading

The CRNN+CTC recognizer has several advantages:

  • Stable training
  • Fast inference
  • Well-understood behavior
  • No need for character-level alignment labels
  • Easy failure analysis with greedy CTC decoding

Limitations:

  • Weak language modeling
  • Ambiguous character predictions
  • Limited context
  • Sensitivity to bad line crops

This made it a good diagnostic model. If it failed badly on clean crops, the recognizer was the problem. If it worked on clean crops but failed on full pipeline outputs, the detector or crop expansion was the bottleneck.

Part 2. Improvements Link to heading

Detection Was Still the Bottleneck Link to heading

After the first end-to-end version, the result was useful but the error analysis pointed back to detection. The recognition model was already sensitive to crop quality: if a crop contained extra text from a neighboring line, or if the target line was cut too tightly, the recognizer could fail even when the text itself was readable.

Many pages were easy. When the lines were straight, well separated, and close to horizontal, the v0 connected-component postprocessor produced good crops. These are the cases where the recognizer receives a clean line image and can do its job.

But the dataset also contained harder layouts:

  • pages with separated blocks of text;
  • curved baselines;
  • text written under a noticeable angle;
  • dense regions where neighboring lines were close;
  • crops where a small amount of extra text changed the recognizer output.

The difficult cases were not always dramatic. Sometimes the oriented box looked reasonable, but the crop still included part of the line above or below. For CTC recognition, this is enough to break the assumption that one crop corresponds to one target sequence.

The conclusion was clear: improving recognition required improving detection. A better recognizer can tolerate some noise, but it cannot reliably choose the intended line when the crop contains multiple plausible text sequences.

Watershed Postprocessing Link to heading

The next postprocessing idea was to treat line extraction as an instance-segmentation problem. The network already predicts two useful maps:

  • line probability: where text centerlines are likely to be;
  • separator probability: where neighboring lines should be split.

The watershed postprocessor converts those maps into line instances. The idea is to create reliable markers from high-confidence line regions, restrict growth to a foreground band around line evidence, and use separator predictions as barriers between competing instances.

 1Line Probability
 2    ↓
 3Threshold + Skeletonize
 4    ↓
 5Markers
 6    ↓
 7Foreground Band
 8    ↓
 9Watershed Elevation
10    ↓
11Line Instances
12    ↓
13Expanded Instance Masks
14    ↓
15Recognition Boxes

In code, the flow is roughly:

  1. Downscale large probability maps for cheaper morphology.
  2. Estimate typical line spacing from row-wise line-probability peaks.
  3. Build markers from confident line-probability regions.
  4. Merge small marker fragments when there is baseline support between them.
  5. Build a foreground mask by expanding around likely text bands.
  6. Create watershed elevation from inverse line probability plus separator penalty.
  7. Run watershed to get line-instance labels.
  8. Extract baselines and line boxes from each instance.
  9. Estimate line height using separator components or local spacing.
  10. Expand instance masks competitively so neighboring lines block each other.

The separator map is important in two places. First, it raises the watershed elevation, making it harder for neighboring line instances to cross separator regions. Second, separator components help estimate line height near baseline endpoints. That gives better crops than relying only on a global line-spacing heuristic.

Compared with v0 connected components, watershed gives a denser interpretation of the page. Instead of starting from sparse superpixels and linking them into curves, it creates competing regions in image space. This is a better fit when line geometry is not cleanly represented by a small set of connected baseline points.

An important follow-up idea was to stop treating the oriented box as the final recognition object. Once watershed gives an instance mask, the crop does not have to be a pure rectangle anymore.

The alternative representation is:

1Watershed Instance Mask
2    ↓
3Axis-Aligned Bounding Box Around Mask
4    ↓
5Crop Image
6    ↓
7Apply Mask Inside Crop
8    ↓
9Recognition Input

This changes the problem. The bbox is only a container used to extract pixels from the page. The mask decides which pixels belong to the line. Extra neighboring text can be suppressed before the recognizer sees the crop.

This is useful because recognition is sensitive to additional text. If a neighboring stroke is inside the rectangle but outside the instance mask, it can be removed or attenuated. The recognizer receives a cleaner visual sequence, while the detector keeps the more flexible geometry of the watershed instance.

The goal was not to make postprocessing more complex for its own sake. The goal was to reduce extra text in recognition crops, because recognition quality was already limited by crop purity.

3. Decoding Link to heading

Why Decoding Matters Link to heading

Even a strong recognizer produces uncertain outputs.

Example:

1market
2markel
3marker

The decoder determines which hypothesis survives.

Beam Search Link to heading

Greedy CTC decoding takes the best character at each time step and then collapses repeats and blanks. It is fast, but it throws away useful alternatives. Beam search keeps several possible prefixes alive while scanning the CTC output, so the final decision can use more than only the locally best visual character.

That matters because the recognizer often produces ambiguous probabilities:

1visual crop -> CTC probabilities -> beam hypotheses
2                                  -> "земля"
3                                  -> "земла"
4                                  -> "земпя"

The visual model proposes several plausible readings. The decoder decides which one survives.

Character-Level RNN Language Model Link to heading

Using only visual information is a lightweight starting point and already gives usable initial results, but it ignores a simple fact: text has structure. Some character sequences are common, some are rare, and some are almost impossible in the target language or domain. Adding this linguistic information during decoding is a standard way to improve CTC-style recognizers. Similar ideas are used in joint CTC/attention speech recognition with an external RNN-LM (Advances in Joint CTC-Attention based End-to-End Speech Recognition with a Deep CNN Encoder and RNN-LM) and in OCR systems such as PyLaia (Improving Automatic Text Recognition with Language Models in the PyLaia Open-Source Library).

For this step I trained a small character-level RNN language model. The model is trained on text only, without images. Given previous characters, it predicts the next character:

1[SOS, c1, c2, ..., cN-1] -> [c1, c2, ..., cN]

The language model is intentionally small: an embedding layer, a two-layer LSTM, dropout, and a linear projection to the character vocabulary. Training uses cross-entropy loss and tracks validation perplexity. The same training setup can build a forward or backward model, but the first integration is the forward RNN-LM used inside CTC beam search.

During decoding, the language model does not replace the recognizer. It only adds a score to each beam expansion:

1score(prefix + c) =
2    CTC_prefix_score(prefix + c)
3  + lambda_lm * log P_lm(c | prefix)

The decoder keeps an RNN hidden state for every active beam. When several beams need the next-character distribution, they are scored together in a batch, so the integration stays cheap. If the recognizer and the LM use slightly different character vocabularies, a small vocabulary mapping is used before adding the LM score.

The practical effect is modest but useful: the visual model still decides what is visible in the crop, while the language model nudges uncertain hypotheses toward more plausible text. It cannot fix a bad line crop or missing characters, and too much LM weight can make the decoder hallucinate common text. For this reason the LM weight is a decoding hyperparameter, not a hard correction rule.

Result after adding RNN-LM rescoring:

1Initial detection + recognition score: 0.746
2With RNN-LM decoding:                 0.794

Beyond CRNN: Conformer Recognizer Link to heading

The baseline CRNN was useful because it was simple and easy to debug, but it is not the only direction for recognition. The field moved from the seminal CNN+RNN+CTC recognizers toward stronger Transformer-style visual sequence models. For handwritten lines this is attractive: a single crop can contain long-range dependencies, local writing habits, abbreviations, and characters whose shape is only clear after looking at neighboring context.

Instead of only replacing the BiLSTM with a larger recurrent stack, I moved to a Conformer-based recognizer, following the general idea from Conformer: Convolution-augmented Transformer for Speech Recognition. The model still starts with the same practical idea as the CRNN: normalize the line crop, extract a compact visual sequence with a CNN, then predict characters over time. The difference is the sequence encoder:

 1line crop
 2    ↓
 3CNN visual features
 4    ↓
 5linear projection + positional encoding
 6    ↓
 7Conformer encoder blocks
 8    ↓
 9CTC head
10    ↓
11Transformer attention decoder

Each Conformer block combines three pieces that are useful for handwriting:

  • feed-forward layers for per-position feature refinement;
  • multi-head self-attention for long-range context across the whole line;
  • depthwise convolution for local shape and stroke-neighborhood patterns.

This gives the recognizer a better balance than a plain LSTM stack. Attention captures global dependencies, while the convolution branch keeps the local inductive bias that is important for character shapes. The implementation also keeps the dense CTC geometry variant, optional spatial transformer, and region-type classification head, so the stronger recognizer still fits the same OCR pipeline.

Guided CTC Training Link to heading

The next issue is how to train this stronger recognizer without losing the deployment advantage of CTC. Attention decoders are powerful because they model output dependencies directly, but autoregressive decoding is slower. CTC is much cheaper at inference time, but its alignment is latent and can be difficult to optimize.

I used the idea from GTC: Guided Training of CTC Towards Efficient and Accurate Scene Text Recognition: an attention-based decoder guides the representation learned by the CTC model. The attention branch sees the same encoder memory and is trained with teacher-forced cross-entropy. The CTC branch is trained with CTC loss on the same line target.

The training objective combines the three losses:

1loss =
2    ctc_weight * CTC_loss
3  + (1 - ctc_weight) * attention_loss
4  + region_type_weight * region_type_loss

The attention decoder pushes the shared encoder toward representations that are easier to align and decode. The CTC head keeps the efficient left-to-right sequence-recognition path. This is the useful compromise: attention helps during training and can also be used for analysis or joint decoding, while CTC remains the lightweight path for deployment.

The result is a recognizer that is still compatible with resource-constrained inference, but no longer depends only on a recurrent sequence model. It can use global context, local stroke structure, and CTC alignment at the same time.

In practice this is the next step after the CRNN baseline: improve recognition quality before adding heavier document-level correction. If the crop is clean, the Conformer+guided-CTC recognizer should make fewer local character mistakes; if the crop is polluted with neighboring text, the same sensitivity remains, which is why detection and masking still matter.

Joint Attention, CTC, and LM Decoder Link to heading

After adding the Conformer recognizer, decoding no longer has to choose between CTC-only and attention-only output. The model produces both:

  • CTC logits over the visual time axis;
  • encoder memory for the Transformer attention decoder.

The decoder uses a joint beam-search scheme for the Conformer model. The same beam keeps the attention hypothesis, the CTC prefix state, and the optional language-model state together.

The beam search is attention-driven: at every step, the attention decoder predicts the next token from the current prefix and the encoder memory. But each candidate token is also checked against the CTC prefix scorer. This is important because attention can model global dependencies, while CTC keeps the prediction anchored to a monotonic left-to-right alignment over the line image.

The combined score for extending a beam is:

1combined_delta =
2    (1 - ctc_weight) * attention_delta
3  + ctc_weight       * ctc_delta
4  + lm_weight        * lm_delta

Where:

  • attention_delta is the Transformer decoder log-probability for the next token;
  • ctc_delta is the change in CTC prefix score after adding that token;
  • lm_delta is the RNN-LM log-probability for the same token, if an LM is enabled.

This gives three complementary signals:

1Attention: "Does this character make sense given the previous output?"
2CTC:       "Can this character be aligned to the visual frames?"
3RNN-LM:    "Is this character sequence linguistically plausible?"

The implementation keeps separate bookkeeping for each beam: the output prefix, CTC prefix state, attention decoder cache, LM hidden state, and accumulated scores. This makes decoding more expensive than greedy CTC, but it is still a local line-level decoder rather than a full document model. The key tunable parameters are ctc_weight, lm_weight, beam width, and token pruning.

In practice this decoder is useful for the stage where recognition errors are no longer only local visual mistakes. CTC prevents the attention branch from drifting away from the image, attention improves sequence modeling, and the language model resolves some ambiguous character choices. Bad crops are still a detection problem, but clean crops benefit from the combined decoder.

Together, the Conformer recognizer, guided CTC training, and joint Attention/CTC/LM decoding improved the final score further:

1After RNN-LM decoding:       0.794
2With joint hybrid decoder:   0.811
